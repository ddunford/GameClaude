# Guide — Unreal Engine behaviour (quirks & gotchas)

> **How the engine *behaves*** — the terrain. (How to *drive* it is `guides/tooling-ue.md`; how the *team* works is `guides/workflow.md`.) This guide is **living**: the moment we hit a quirk, we verify it and write it here, same session (doctrine 13).

## The rule for every entry
Each fact carries a **provenance tag** — no unverified claim is trusted:
- `[SRC: path:line]` — an engine-source citation exists; checkable any time. Strongest.
- `[MEASURED: date]` — observed live in this engine build. Proof for runtime behaviour.
- `[web]` — from Epic docs / a reputable source (URL). A **claim**, not proof — verify against source before *building on it* (doctrine 10).
- `[verify]` — believed true, not yet confirmed. Resolve before it's load-bearing.

Citations tagged `[SRC:…]` were verified during trial-1; they point at real source lines and are checkable, but a line may have shifted across engine updates — re-confirm if a fact turns load-bearing. The **re-verify queue** at the bottom lists the load-bearing facts a web pass could not independently confirm — `engine-verifier`'s worklist.

## 1. Engine version & targets
- Target is **UE 5.8.0**. `[verify]` (project rule)
- **A dedicated-server target needs a source-built engine; a Launcher/EGS binary install cannot link it** — the server-flavoured libs aren't shipped, and no plugin substitutes. `[web]` [Epic — Setting Up Dedicated Servers](https://dev.epicgames.com/documentation/unreal-engine/setting-up-dedicated-servers-in-unreal-engine) · corroborated `[SRC: PlayLevel.cpp:2418]`
- **PIE ▸ Play As Client launches a real in-process dedicated-server world** (`NM_DedicatedServer`) — true server authority for *correctness* without a server target. `[SRC: PlayLevel.cpp:2418]` · `[web]` [Epic — PIE Multiplayer Options](https://dev.epicgames.com/documentation/unreal-engine/play-in-editor-multiplayer-options-in-unreal-engine)
- **`UnrealEditor.exe -server` runs a genuine out-of-process dedicated server** and Iris drives its netdriver. `[MEASURED: 2026-07-19]`
- Both PIE-as-client and `-server` run **`WITH_EDITOR=1`** and leave cooked-server paths untested: valid for authority/replication/relevancy correctness, **invalid for performance numbers** (frame time, bandwidth). `[verify]`
- **`Build.bat` for editor/client targets must run with the editor CLOSED** — a running editor holds the binary locks. `[MEASURED: 2026-07-19]`

## 2. World Partition & streaming
- Server streaming is **opt-in and off by default**: `wp.Runtime.EnableServerStreaming=1` makes the server stream around player sources instead of loading the whole map. Without it the server loads everything (CPU pinned in production). `[web]` [Epic KB — WP Server Streaming](https://forums.unrealengine.com/t/world-partition-server-streaming/688971)
- **`wp.Runtime.EnableServerStreamingOut` stays `0`.** Streaming a cell OUT destroys its actors — on an authoritative server that is data loss. `[SRC: architecture.md §7]` ⚠ **re-verify** (the "streaming-out = actor destruction" causal claim is web-unconfirmed). Two mechanics of leaving it `0` *are* cited: the server first activates all non-data-layer cells (`WorldPartitionStreamingPolicy.cpp:442-454`), then its loaded set is **monotonically increasing** — every already-loaded cell is re-added every frame (`:506-517`), so residency converges on the **union of every cell any player has visited since start-up**. Server streaming buys *deferred* loading, not *bounded* residency: budget server memory against the **full map**, and treat a restart as the only reset. Also: the server will not deactivate a cell while any client still has it visible, so one AFK player pins it for everyone (`:522-537`). Interim lever to pin only stateful layers resident: `wp.Runtime.ServerDisallowStreamingOutDataLayers` (a comma-separated allowlist, `:46-51`). Prefer the per-map `EWorldPartitionServerStreamingMode` in World Settings (`WorldPartition.h:71-86`) over the global cvar — source-controlled and immune to ini mistakes. Re-enable streaming-out only once persistence-on-unload exists. `[SRC: WorldPartitionStreamingPolicy.cpp:442-454,506-517,522-537]`
- **WP grid cell size + loading range are UI-only in practice** — `UWorldPartition::RuntimeHash` is `protected` and `AWorldSettings::WorldPartition` is `VisibleAnywhere`, so MCP/Python can't set them; it's a World Settings ▸ World Partition Setup action. `[MEASURED: 2026-07-19]` · `[web]` [Epic — World Partition](https://dev.epicgames.com/documentation/en-us/unreal-engine/world-partition-in-unreal-engine)
- Defaults kept consciously: **12800 uu cells / 25600 uu loading range**. Loading range is a radius → doubling it is ~4× loaded set/memory. `[verify]`
- **Changing cell size later re-buckets every actor and forces a full HLOD rebuild.** `[verify]`
- **HLODs are a build step, not automatic** (Build ▸ Build HLODs, or the WP HLODs commandlet) — and not currently triggerable from the agent toolset. `[web]` [Epic — World Partition](https://dev.epicgames.com/documentation/en-us/unreal-engine/world-partition-in-unreal-engine)
- **HLOD layer assets clone from `/Engine/Maps/Templates/HLODs/HLODLayer_Instanced`/`_Merged`; repoint the clone's `parentLayer` at the local copy** or it keeps an engine dependency. A map from the OpenWorld template silently points `default_hlod_layer` at `/Engine/OpenWorldTemplate/HLOD/HLOD0_Instancing` — repoint it. `[MEASURED: 2026-07-19]`
- **`wp.Runtime.HLOD.ForceDisable` does NOT exist in 5.8** (a widespread community claim). The real control is the `wp.Runtime.HLOD` console *command* (`HLODRuntimeSubsystem.cpp:147-192`, default on) — being a command it **cannot be set from ini**; use `-ExecCmds="wp.Runtime.HLOD 0"`. The server **does** load HLOD cells (`ShouldSkipDisabledHLODCell` has no `bIsServer` term, `:540-543`, while the perf-skip directly below it does). ⚠ Whether HLOD proxy meshes carry server-side collision is unverified. `[SRC: HLODRuntimeSubsystem.cpp:147-192,540-543]`

### Data Layers — the client/server split, and why not for districts
- **Data Layers are NOT a district mechanism** — `AWorldDataLayers` is `bAlwaysRelevant = true`, so every client receives the full state of every layer regardless of distance; a layer-per-district scales the wrong way. Keep runtime layer count in the **tens**. Districts are **spatial** (WP cells + streaming sources handle them for free); the per-actor levers are `RuntimeGrid` (an `FName` on `AActor`) and `bIsSpatiallyLoaded`. `[SRC: WorldDataLayers.cpp:54-78]`
- **Query a layer's *effective* state, not its requested state** — four replicated arrays exist: `Loaded`/`Active` are requested, `EffectiveLoaded`/`EffectiveActive` are resolved after the parent hierarchy. `[SRC]` (architecture.md §6)
- **Data Layer load filters give client-only / server-only content, enforced in engine:** `EDataLayerLoadFilter { None, ClientOnly, ServerOnly }` (`DataLayerAsset.h:17-26`); the server is *rejected* from changing a `ClientOnly` layer (`ClientOnlyFromServer`, `WorldDataLayers.cpp:103`) and the client from a `ServerOnly` one (`WorldDataLayers.cpp:84-119`). `ClientOnly` → cosmetics, signage detail, ambient VFX, client crowds/traffic: zero server memory, physics and replication — **the mechanism behind "ambient life is client-side only".** `ServerOnly` → spawn logic, AI nav, anti-cheat volumes, district-ruleset-enforcement volumes clients never receive (a "never trust the client" win). Requires `IsRuntime()` — an Editor-type layer silently ignores the filter; `InitialRuntimeState` is unsupported for client/server-only layers (`DataLayerInstance.h:237`). `[SRC: DataLayerAsset.h:17-26; WorldDataLayers.cpp:84-119]`
- **A load filter is authored on the ASSET but only ever consulted through an INSTANCE.** `LoadFilter` lives on `UDataLayerAsset` (`DataLayerAsset.h:87-88`, `IsClientOnly()` at `:70`), but streaming reads it via a `UDataLayerInstance` (`UDataLayerInstanceWithAsset::IsClientOnly()` forwards to the asset, `DataLayerInstanceWithAsset.h:46`; the base returns `false`, `DataLayerInstance.h:157`). **An asset assigned to an actor with no `DataLayerInstance` in that world does nothing — the actor loads everywhere.** Each world needs its own instance. `[MEASURED: 2026-07-18]` · `[SRC: DataLayerAsset.h:70,87-88; DataLayerInstance.h:157]`
- **`ClientOnly` layers do not load on a dedicated server** — verified on a live `-server` run: a `ClientOnly` marker *with an instance* was absent from `obj list`, the same asset *without an instance* present. `[MEASURED: 2026-07-18]` ⚠ **A LISTEN server with server streaming still loads client-only cells** (`WorldPartitionRuntimeHash.cpp:615-620`) — the filter is not a security boundary on a listen host (Epic treats the area as unfinished: `//@todo_ow` at `WorldPartitionRuntimeCell.h:357`). Harmless here only because listen servers are never used for authority work.

## 3. Replication / Iris
- **Iris status, 5.8:** the plugin metadata still reads Beta (`Iris.uplugin`: `IsBetaVersion:true`, `VersionName:0.1`, `EnabledByDefault:false`) `[SRC: Iris.uplugin]`, while the **5.8 release notes call Iris "production-ready for licensees"** `[web]` [UE 5.8 Release Notes](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-5-8-release-notes). Net: usable and promoted, but Beta still permits breaking API changes across upgrades — don't claim packaged-build correctness.
- **Enabling Iris in 5.8 is a 2-place RUNTIME model, not a build-flag one:** (1) the **Iris plugin** enabled in `.uproject` (loads `IrisCore`), and (2) two cvars in `[SystemSettings]` — `net.Iris.UseIrisReplication=1` **and** `net.SubObjects.DefaultUseSubObjectReplicationList=1`. The net driver decides at init from these — `WillNetDriverUseIris` reads the cvar, gated by the plugin being loaded (`UnrealEngine.cpp:14798-14886`), and the cvar defaults off in `BaseEngine.ini:346`. **There is NO `bUseIris` `TargetRules` flag in UE 5.8 — it does not exist, and adding it to a `*.Target.cs` breaks the UBT build.** `SetupIrisSupport(Target)` in a game module's `.Build.cs` is **optional** — needed only when *that module* declares its own FastArray serializer; a project using engine/GAS replicated types needs neither build edit. `[SRC: UnrealEngine.cpp:14798-14886; BaseEngine.ini:346]` · `[MEASURED: 2026-07-21]` (engine-verifier + live boot log)
- **`DefaultUseSubObjectReplicationList=1` is mandatory** — the engine asserts without it. `[SRC: EngineReplicationBridge.cpp:234]` · corroborated `[web]` (Migrate to Iris: subobject list is a hard dependency)
- **Every `AActor` is spatially filtered by DEFAULT; do NOT add `FilterConfigs` to "enable" relevancy** — they *override* the default. Add entries only to make a class always-relevant/unrouted. `bAlwaysRelevant` actors (PlayerState, GameState) are **exempted** from the spatial filter, so they reach every client regardless of distance. `[SRC: EngineReplicationBridge.cpp:243-253]`
- **GAS has first-class Iris support in 5.8** — the engine's `GameplayAbilities` module calls `SetupIrisSupport` itself (`GameplayAbilities.Build.cs:42`) and ships a full NetSerializer suite for GAS types, so an `AbilitySystemComponent` replicates correctly under Iris with **no game-side build wiring**. The tag-count-container fallback path is gated by a cvar that **defaults false** (`GameplayEffectTypes.cpp:47-51`) — the NetSerializer path is the default. Use `EGameplayEffectReplicationMode::Mixed` on a player-owned ASC (full replication to the owning client, minimal to others). `[SRC: GameplayAbilities.Build.cs:42; GameplayEffectTypes.cpp:47-51]`
- **ASC-on-PlayerState reaches all clients under Iris** — `APlayerState` is `bAlwaysRelevant = true` (`PlayerState.cpp:25-26`), which exempts it from the default Iris spatial filter (`EngineReplicationBridge.cpp:243-253`); so an ASC hosted on PlayerState, and its tags/cues, replicate to every client regardless of distance — the correct home for a persistent, player-owned kit. `[SRC: PlayerState.cpp:25-26; EngineReplicationBridge.cpp:243-253]`
- **Spatial separation is plain distance culling** — per-object cull distance (default 150 m 3D); keep zone actors under `MaxNetCullDistance` or they go always-relevant. `[SRC: NetObjectGridFilter.cpp:211,494-500; Actor.cpp:312,101,298]`
- **Live-verified separation at a 2 km offset** (server + 2 clients, Character proxy counts): both at hub = 6; one moved 2 km = 4 (far pawn culled from both other views); both at the offset = 6. `[MEASURED: 2026-07-20]`
- **Iris hard-caps at 128 connections; unchangeable on a Launcher build** — `MaxConnections=128` is a private-header default-arg with no setter/cvar/ini; raising it needs engine source. `[SRC: ReplicationConnections.h:40-45]`
- **Push model is PIE-only on installed builds** (`WITH_PUSH_MODEL=0` in Game/Server targets) — write as if on, don't claim packaged-build push correctness. It's opt-in per property (`DOREPLIFETIME_WITH_PARAMS_FAST` + `MARK_PROPERTY_DIRTY`); miss the mark and it silently reverts to polling. `[SRC: WITH_PUSH_MODEL=0]` · `[web]` (StraySpark, community)
- **Push model is change-DETECTION, not the on-wire delta.** Delta compression — sending only the properties changed since the last acked baseline — is done by the core replication machinery *with or without* push model (`PushModel.h:24-25`). Push model changes only how dirtiness is **detected**: instead of comparing every replicated property against its last-sent value each cycle (`PushModel.h:27-38`), gameplay code calls `MARK_PROPERTY_DIRTY` so clean properties are skipped and only dirtied ones are compared (`PushModel.h:95-115`). It is a server-CPU optimization, not a change to what goes over the wire. `[SRC: PushModel.h:24-25,27-38,95-115]`
- **Per-connection bandwidth default is ~10 KB/s.** `MaxInternetClientRate=10000` bytes/s (`NetDriver.cpp:669`) caps `MaxClientRate` down from 15000 for internet play (`World.cpp:7987-7989`); each connection is then clamped to `[1800, MaxClientRate]` (`PlayerController.cpp:539`). Budget the per-actor share against this per-connection cap × 128 connections, not against what localhost PIE tolerates. `[SRC: NetDriver.cpp:669; World.cpp:7987-7989; PlayerController.cpp:539]`
- `LevelStreaming` Level Instance cheap replication path: `COND_InitialOnly`. `[SRC: LevelInstanceActor.cpp:87-95]`
- **Level Instance type display-names are SWAPPED:** `Partitioned` shows as **"Embedded"**, `LevelStreaming` shows as **"Standalone"** (`LevelInstanceTypes.h:55-66`) — docs use the UI names, code the identifiers. **`Partitioned` costs zero at runtime** — streaming generation dissolves it and hoists its children into the main grid (`LevelInstanceActor.cpp:222-226`); use it liberally for authoring grouping. Its free-ness gate is **OFPA**, and failure is silent: a non-OFPA source level falls back to level streaming regardless of the dropdown (`LevelInstanceActorDesc.cpp:219-243`) — add a validation check. A **PLA Blueprint** stores **zero** per-instance transform data across N placements (`bInheritPerInstanceData=true`, `PackedLevelActorBuilder.cpp:453-491`). Only `LevelStreaming` gives a real per-instance streaming level — the dynamic-zone / door path. `[SRC: LevelInstanceTypes.h:55-66; LevelInstanceActor.cpp:222-226; LevelInstanceActorDesc.cpp:219-243]`
- **Runtime-spawned actors belong to NO World Partition cell — you spawn it, you destroy it.** WP cell membership is a *cooked* mapping of asset paths (`FWorldPartitionRuntimeCellObjectMapping`) and cell-mutation APIs are `#if WITH_EDITOR` (`WorldPartitionRuntimeCell.h`); **WP will never unload player-built / runtime-spawned content.** Use `ULevelStreamingDynamic::LoadLevelInstance` for a zone *template*, never one level instance per prop. `[SRC: WorldPartitionRuntimeCell.h]`
- **You cannot replicate a placement array** — `net.MaxRepArraySize` caps at 2048 elements / 65535 bytes, and `PerInstanceSMData` carries no `Replicated` specifier so instance arrays never go over the wire on their own (`InstancedStaticMeshComponent.h:168`). Serve large placement sets **out-of-band** (HTTP, keyed by a version hash) and replicate only the hash plus a push-model fast-array of **deltas** on a **dormant** actor (Epic's own topology in `InstancedActors`: `InstancedActorsManager.cpp:217-218` sets `bReplicates=true; SetNetDormancy(DORM_DormantAll)`, deltas at `InstancedActorsData.cpp:1059` — copy the topology, do not adopt the experimental plugin). `[SRC: InstancedStaticMeshComponent.h:168]`
- **Editor `NoRedist` toolsets run Python startup server-side under `-server`**, after the replication system starts → Iris warns `BroadcastLoadedModulesUpdated() called while there are 1 active ReplicationSystems` (risk of corrupt polymorphic data). Disable editor toolsets for `-server` runs. `[MEASURED: 2026-07-19]`

## 4. Collision
- **ISM collision is per-COMPONENT, not per-instance** — all instances share one collision setting, so there's no runtime per-instance toggle. Set `NoCollision` per component in the PLA source level **before packing**. `[web]` [Epic — Instanced Static Mesh Component](https://dev.epicgames.com/documentation/en-us/unreal-engine/instanced-static-mesh-component-in-unreal-engine)
- **A packed kit creates ISM collision bodies on the dedicated server with no server check** — `OnCreatePhysicsState` calls `CreateAllInstanceBodies()` with no `IsRunningDedicatedServer()` anywhere in the path; rendering is free on a server (`FApp::CanEverRender()` is false under `UE_SERVER`) but collision is not, at one `FBodyInstance` per instance. This is why the above must happen pre-pack. `[SRC: InstancedStaticMesh.cpp:2908-2925]`
- Measured server cost of one packed block: **1,826 ISM components + 665 BodySetups** (BodySetup is per mesh asset, so it understates per-instance cost). `[MEASURED: 2026-07-19]`
- **UE 5.8 collision channels:** ~6 default object + 2 default trace, plus up to **18 CUSTOM** channels — a hard ceiling to budget. (Trial-1 measured 50 total slots including built-ins; the *custom* budget is 18.) `[web]` [Epic — Add a Custom Object Type](https://dev.epicgames.com/documentation/unreal-engine/add-a-custom-object-type-to-your-project-in-unreal-engine) · `[MEASURED: 2026-07-19]`
- **Channels are integer indices — inserting/reordering silently reassigns every asset that referenced a shifted index.** Reserve append-only. `[verify]`
- **Collision config (`[/Script/Engine.CollisionProfile]`) is hand-edited in `DefaultEngine.ini` and read at startup** (restart to apply). Opening Project Settings ▸ Collision first rewrites the section from in-memory state and wipes the edit; `ConfigSettingsToolset` can't reach it (custom Slate widget). `[MEASURED: 2026-07-19]` ⚠ **re-verify** (rewrite-on-open is project lore)
- **A missing collision profile silently reverts** when set on a component — read-back after setting is the only proof. `[MEASURED: 2026-07-19]`

## 5. Lighting / rendering
- **Sky Atmosphere, Directional Light, Sky Light, height fog, volumetric clouds are scene-global — one active set per world.** Sky Atmosphere supports at most **2 atmospheric directional lights**. `[web]` [Epic — Sky Atmosphere Component](https://dev.epicgames.com/documentation/en-us/unreal-engine/sky-atmosphere-component-in-unreal-engine)
- **A `LevelStreaming` Level Instance streams into the persistent world's ONE `FScene`, so it shares the global day sky** — a Level Instance does NOT grant an independent sky. `[SRC: World.h:1504; SkyAtmosphereRendering.cpp:690-695; RendererScene.cpp:4710-4716; FogRendering.cpp:104]`
- So a night region under a shared day sky is built **spatially**: full enclosure + an **overhead occluder** (an open-top area still sees the global sky and is lit by the global sun) + a **bounded manual-exposure `PostProcessVolume`** + local lights. The occluder must actually seal — if the lid sits above the walls, the day sky leaks the seam and the global sun/skylight still light the interior. `[SRC: architecture.md §7]` · `[MEASURED: 2026-07-20]`
- **Supported baked day/night pattern = Lighting Scenario levels** (one visible at a time; each register's lightmaps bake separately). For a Lumen/dynamic day-night, animate the single shared sun/sky instead. `[web]` [Epic — Precomputed Lighting Scenarios](https://dev.epicgames.com/documentation/unreal-engine/using-precomputed-lighting-scenarios-in-unreal-engine)
- **Use MANUAL exposure, locked per register** — auto-exposure re-balances to whatever's in frame (produced an "olive road cast" from a bright-sky-tuned bias). But **locked ≠ crushed**: exposure locked dark to fake night rendered a fully-planted garden (18 trees + 200+ plants) near-black. Stay visible. The "bias" number is empirical, not EV semantics (11 blew white, 4 read night here). `[MEASURED: 2026-07-19 / 2026-07-20]`
- **Packed Level Actors silently drop everything but static meshes and nested level instances** — only two builders exist, ISM and recursive (`PackedLevelActorBuilder.cpp:127-147`), and the ISM builder clusters **only `UStaticMeshComponent`s** (`PackedLevelActorISMBuilder.cpp:37-48`); skeletal meshes, **lights**, decals, **Niagara**, **audio components (`AmbientSound`/`UAudioComponent`)**, splines, volumes and **all Blueprint logic** take the "Component was not packed" warning path (`PackedLevelActorBuilder.cpp:255-276`) and are dropped. So door light-spill / neon inside a packed building must be **emissive material**, not a light or Niagara; and interior point audio must be authored as a **separate external World-Partition actor** — exactly the treatment door light-spill / neon gets — never inside the PLA. `[SRC: PackedLevelActorBuilder.cpp:127-147,255-276; PackedLevelActorISMBuilder.cpp:37-48]`
- **A Nanite mesh in a PLA becomes a plain ISM, not HISM** (`PackedLevelActorISMBuilder.cpp:70-78`) — correct, since Nanite has its own culling/LOD, though it contradicts Epic docs. Note **Nanite reduces per-draw GPU cost, not draw calls** — instancing is still required. `[SRC: PackedLevelActorISMBuilder.cpp:70-78]`
- **A stale, un-resaved PLA re-runs its construction scripts at load — on the server** (`PackedLevelActor.cpp:65-98`). Grep server logs for the version-mismatch message in CI. `[SRC: PackedLevelActor.cpp:65-98]`
- Baseline is **Lumen + Nanite**. Thin-wall light leak is a Lumen mesh-distance-field resolution problem (thin walls vanish from the SDF). `[verify]`
- **A freshly created empty level (`new_level`) has no sky / sun / skylight, so a SceneCapture FinalColor render comes back fully BLACK.** Add a `DirectionalLight` + `SkyAtmosphere` + `SkyLight` before capturing anything. `[MEASURED: 2026-07-21]` (smoke test)
- **`USceneCaptureComponent2D` does true orthographic capture** — set `projection_type = CameraProjectionMode.ORTHOGRAPHIC` and `ortho_width` (cm), render to a `TextureRenderTarget2D`, export with `RenderingLibrary.export_render_target`. Confirmed genuinely parallel: the image is **pixel-identical across camera distances** (mean diff 0.0 between camera Z 2500 and 4000), whereas perspective changes with distance (diff 1.4–21). Identical cubes at different depths project to identical size. This is the only capture path that gives a non-foreshortened plan/elevation. `[MEASURED: 2026-07-21]`
- **A single directional light cannot light an orthographic *elevation*** — the faces pointing at the camera get no direct light and crush to black at the exposure that suits the sun-lit faces (a full row of cubes rendered pure black while base-colour proved them in-frame). A capture/legibility rig needs a **key + opposing fill directional pair (shadow-casting off) + a skylight**, so every face of an axis-aligned mass catches direct light from one of them. `[MEASURED: 2026-07-21]`
- **A cold / freshly-reposed `SceneCapture2D` renders black on its first `capture_scene()`** — call it several times before exporting (the toolkit primes ~6×). `[MEASURED: 2026-07-21]`

## 6. PIE & dedicated server
- **Net mode is NOT a `StartPIE` parameter** — the engine reads it from the `ULevelEditorPlaySettings` CDO (`/Script/UnrealEd.Default__LevelEditorPlaySettings`): set `playNetMode` / `playNumberOfClients` / `runUnderOneProcess`, then StartPIE. `[SRC: PlayLevel.cpp:991]`
- **The CDO is global editor state — record prior values, restore after the run.** A leak silently changes every later test's net mode. `[verify]`
- **`runUnderOneProcess` inverts:** ON for agent runs (in-process; the watcher subscribes to `PostPIEStarted`); OFF for manual multi-client testing (more representative). `[verify]`
- **Never use Play As Listen Server for authority work** — the host *is* the server, so it can't test "never trust the client". `[web]` [Epic — PIE Multiplayer Options](https://dev.epicgames.com/documentation/unreal-engine/play-in-editor-multiplayer-options-in-unreal-engine)
- **One client can't verify replication** — client 2 catches owner-relevancy / `bOnlyRelevantToOwner` mistakes. A *single* client under a dedicated-server config can misreport `NM_Standalone`; use ≥1 client with the server explicitly enabled. `[web]` (Epic forum) · `[verify]`
- **Player 2 spawns at world origin (0,0,0) with no 2nd PlayerStart** and falls through thin scaled-cube ground — need proper PlayerStarts + solid ground before a real multiplayer walk. `[MEASURED: 2026-07-20]`
- **CQTest** (`PIENetworkComponent.h`) is a networked-PIE harness defaulting to `bIsDedicatedServer=true`, drivable via `AutomationTestToolset`; needs `"CQTest"` in the module `.Build.cs`. `[SRC: PIENetworkComponent.h]` It is the CI-grade C++ route; the Python `NetworkHarnessTools` rig (below) is the live-now route that needs no build/restart and works against the stock template before any game character exists.

### Driving multi-client PIE from Python — the harness recipe (`NetworkHarnessTools`)
- **The `LevelEditorPlaySettings` CDO props are NOT exposed to Python `get_editor_property`** (it is an editor-config class; props aren't BlueprintReadWrite — `dir()` shows none). Read/write them via **`unreal.ToolsetLibrary.get_object_properties(cdo, [...])` / `set_object_properties(cdo, json_str)`** (the same raw-FProperty reflection Epic's `ObjectTools` uses). CDO object: `unreal.load_object(None, "/Script/UnrealEd.Default__LevelEditorPlaySettings")`. Keys are camelCase: `playNetMode` (`PIE_Standalone`|`PIE_ListenServer`|`PIE_Client`), `playNumberOfClients`, `runUnderOneProcess`. `[MEASURED: 2026-07-21]`
- **Start/stop PIE from Python** with `unreal.LevelEditorSubsystem.editor_request_begin_play()` / `editor_request_end_play()`; it honours the CDO net mode. Both are **async** — PIE is not up the instant begin returns; poll `is_in_play_in_editor()` and world enumeration. `[MEASURED: 2026-07-21]`
- **A tool call cannot start PIE and then drive/assert in the same call** — MCP/`py` calls run on the game thread, which is where PIE ticks, so a blocking call starves the tick. A PIE harness must be **stateful across calls**: begin → (editor ticks between calls) → read/drive/assert → end. `[MEASURED: 2026-07-21]`
- **With `runUnderOneProcess=True`, server + all clients live in this process and are reachable by deterministic path:** for base map `/Dir/Map.Map`, PIE instance N is `/Dir/UEDPIE_{N}_Map.Map`, fetched with `unreal.load_object(None, path)` (returns the in-memory renamed world, no disk reload). **Instance 0 is the dedicated server; 1..N are the clients.** There is **no generic UWorld iterator** in Python (`find_all_objects` etc. don't exist) and `UnrealEditorSubsystem.get_game_world()` returns only one world — deriving paths is the way to reach all of them. `[MEASURED: 2026-07-21]`
- **Server vs client is told apart by `GameplayStatics.get_game_mode(world)`** — non-null only on the authority (server); `None` on clients. `UWorld` has no `get_net_mode()` Python binding. `[MEASURED: 2026-07-21]`
- **A pawn's local UObject name (`…_C_0` / `_C_1`) is NOT a stable cross-world key** — the same physical player is `_C_0` in one world and `_C_1` in another. Match a pawn across worlds by **proximity to a known position**, not by name. `[MEASURED: 2026-07-21]`
- **A server-authoritative `set_actor_location` on the instance-0 pawn replicates to every client world within a few ticks** (proven: server moved a pawn to (2000,-1200), both clients showed it within ~35 cm after settling). So the replication assertion is: move on the server, let the editor tick, then assert every client sees a pawn there. `[MEASURED: 2026-07-21]`
- **Iris is live on the `GameNetDriver`** in both PIE and `-server` — boot logs show `LogNet: InitBase GameNetDriver (NetDriverDefinition GameNetDriver) using replication model Iris`, and one `Iris ReplicationSystem` is created per world. It is NOT silently falling back. The enablement model (Iris plugin + the two `[SystemSettings]` cvars; no `bUseIris` build flag) is in §3. `[MEASURED: 2026-07-21]` (engine-verifier pass + live boot log)
- **`UnrealEditor.exe -server` on a Launcher build boots a genuine dedicated-server world** — log shows `World Type = Game, IsEditor = 0`, `IpNetDriver listening on port <P>` / `Created socket for bind address: 0.0.0.0:<P>`, and Iris on the net driver. No server *target* (which won't link on a Launcher engine) is needed for correctness/feel. `[MEASURED: 2026-07-21]`

## 7. Assets (bounds lie, async-load, import)
- **`get_actor_bounds()` on a Packed Level Actor reports the packed asset's ORIGIN, not the rendered geometry** — it read base `-128` for 27 buildings floating 3.25–12.75 m, and three "floating=0" checks passed on a visibly broken level. Measure true base per-ISM: `true_z = t.translation.z + (mb.origin.z − mb.box_extent.z) * t.scale3d.z`. `[MEASURED: 2026-07-19]`
- **When two measurements of one actor disagree, the contradiction IS the finding** — don't pick the convenient one. `[MEASURED: 2026-07-19]`
- **A suspiciously small bound (~2.6 m) is an async-load artifact, not a small asset** — re-query anything implausible. `[MEASURED: 2026-07-19]`
- **A raw obj/gltf in metres imports 100× oversize in cm** (a fountain at 250 m) — place at scale 0.01 if not baked. Free CC BY "scans" arrive raw/oversized/mis-oriented (a bollard 98k tris/11 m; a shutter 4.2M tris/21 m). `[MEASURED: 2026-07-19 / 2026-07-20]`
- **`Texture2D.max_texture_size` caps RUNTIME resolution only** — the full-res source still sits in the `.uasset` (a "2K-capped" texture was 29 MB on disk). Resize the source to shrink the repo. `[MEASURED: 2026-07-19]`
- **Decimating a scan strips UVs → texture stops mapping.** Enable Nanite on the full mesh instead (preserves UVs, swallows the triangles). `[MEASURED: 2026-07-19]`
- **`AssetTools.duplicate` does NOT rewrite references** — a duplicate still points at the vendor pack. Use Migrate / `migrate_packages` via the Remote Control Python route to curate. `[verify]`

## 8. Config (which section, silent-ignore)
- **Console variables set from ini MUST go in `[SystemSettings]` in `DefaultEngine.ini`** (`[SystemSettingsEditor]` for editor-only). In any other section they are **silently ignored** — no error. The single most common "my cvar has no effect" cause. `[web]` [Epic — Console Variables in C++](https://dev.epicgames.com/documentation/unreal-engine/console-variables-cplusplus-in-unreal-engine) · `[SRC: BaseEngine.ini:2464]`
- Distinct from `Engine/Config/ConsoleVariables.ini`, which has its own `[Startup]` section for early cvars — don't confuse them. `[web]`
- The `[SystemSettings]` block in use: `net.Iris.UseIrisReplication=1`, `net.SubObjects.DefaultUseSubObjectReplicationList=1`, `wp.Runtime.EnableServerStreaming=1`, `wp.Runtime.EnableServerStreamingOut=0`. `[verify]`
- A cvar's `ECVF_` flags say when it can be set — **`ECVF_Runtime_ReadOnly` = startup-only.** `EditorAppToolset.SearchCVars` returns the *live* value (proves config actually took). `[verify]`

## 9. Travel between worlds (doors & zone transitions)
- **One `WorldContext` per `GameEngine` in shipping builds — there is no in-process multi-world path.** Running N copies of a world means N processes plus an orchestrator. `[SRC]` (Epic engine comment)
- **Cross-process (server-to-server) travel works, but is never seamless.** `TRAVEL_Absolute` exists for it (`EngineBaseTypes.h:765`, commented *"Traveling from server to server."*); the non-seamless path calls `GEngine->SetClientTravel` (`PlayerController.cpp:5746-5754`). It is a disconnect + reconnect + full map load — a loading screen exactly where the player expected to keep walking. `[SRC: EngineBaseTypes.h:765; PlayerController.cpp:5746-5754]`
- **Seamless travel is same-server + local-package only** — `UWorld::SeamlessTravel`'s map name *"must be relative to the current URL (same server)"* (`World.cpp:9132`) and `FSeamlessTravelHandler::StartTravel` hard-requires `FPackageName::DoesPackageExist(MapName)`, i.e. a local package (`World.cpp:8274-8291`). This is why a same-server door (a `LevelStreaming` Level Instance at a large offset in one world) is the seamless path and a cross-district boundary is not — put process boundaries only at chokepoints (metro/lift/tunnel/door), never at a street. `[SRC: World.cpp:9132,8274-8291]`
- **A same-world door separates its two spaces by ordinary net distance-culling, not special machinery** (see §3) — a zone offset ~2 km from the hub falls outside every hub player's relevancy AABB and vice versa. ⚠ **The destination must be always-loaded on the server or the player falls through on arrival** — with server streaming on, a zone cell streams in with a lag and a teleported pawn arrives before the floor exists; set the destination actors `bIsSpatiallyLoaded = false`. `[MEASURED: 2026-07-20]`
